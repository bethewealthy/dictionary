# 05 · 검증기 계약

> 검증 결과가 편집자·실행 시점에 따라 달라지면 검증기는 쓸모가 없다.
> 이 문서는 **무엇을 단어로 셀 것인가**를 애매함 없이 고정한다.
> 규칙의 근거는 [01-content-policy](01-content-policy.md), 데이터 구조는 `src/types/entry.ts`.

## 단일 원천

| 대상 | 원천 | 읽는 주체 |
|---|---|---|
| 정의 어휘 1,000단어 | `data/defining-vocabulary.json` | **검증기만** |
| 불규칙 굴절형 매핑 | `data/inflections.json` | 검증기만 |
| 표제어 | `data/entries.json` | 앱 + 검증기 |

앱 런타임은 정의 어휘를 알 필요가 없다. 그래서 `Dictionary` 타입에서 `definingVocabulary` 필드를 제거했다.

---

## 1. 토큰화 명세

정의문과 예문을 검사하기 전에 아래 순서로 토큰을 만든다. **순서가 결과를 바꾸므로 순서까지 규칙이다.**

### 1) 마커 제거

- `{{ }}` (예문의 표제어 표시) → 안쪽 내용만 남긴다
- `[[표기|entryId]]` (정의 내 링크) → **표기만 남기고, entryId는 링크 검증으로 따로 넘긴다**

### 2) 분리

- 공백으로 자른다
- 앞뒤 문장부호 제거: `. , ! ? ; : " ' ( )`
- **하이픈은 자르지 않는다.** `ice-cream`은 한 토큰이며 목록에 그대로 등재한다
- **아포스트로피는 단어의 일부다.** `don't`는 `do` + `not`으로 쪼개지 않고 `don't` 자체를 목록에 등재한다
  - *왜*: 축약형 분해 규칙(`'s`=is/has/소유격)은 문맥 판단이 필요해 자동화가 불안정하다. 목록에 몇십 개 더 넣는 쪽이 결정적이다
  - **예외**: 소유격 `'s`는 분리한다. `my sister's book` → `sister`, `book`
- 숫자만으로 된 토큰은 검사에서 제외

### 3) 정규화

- 소문자로 변환
- **문장 첫 단어는 항상 소문자화한다**
- **문장 첫 단어가 아닌데 대문자로 시작하면 고유명사로 보고 검사를 면제한다**
  - 이 규칙 때문에 편집자가 문장 중간에서 실수로 대문자를 쓰면 검사를 빠져나간다. 감수한다 — 반대 방향(고유명사를 매번 목록에 등재)이 훨씬 비싸다

### 4) 원형 환원

굴절형은 원형으로 되돌린 뒤 목록과 대조한다.

- **불규칙형**: `data/inflections.json` 테이블로 환원 (`ran`→`run`, `teeth`→`tooth`)
- **규칙형**: 접미사 규칙으로 환원 (`-s`, `-es`, `-ed`, `-ing`, `-er`, `-est`)
- 환원 결과가 목록에 없으면 **환원 전 형태로 한 번 더** 대조한다 (`glasses`가 목록에 독립 등재된 경우)

### 5) 구동사는 특별 취급하지 않는다

`look after`는 `look`, `after` 두 토큰으로 각각 검사한다. 구동사 단위 검사는 다의성 판단이 필요해 검증기 범위를 벗어난다. 구동사의 뜻은 `Collocation`으로 다룬다.

## 2. 정의 어휘 검사

`senses[].en`의 모든 토큰이 정의 어휘 목록에 있어야 한다.

### 면제 대상

1. 표제어 자기 자신과 그 굴절형
2. 고유명사 (토큰화 3단계에서 판별)
3. `[[ ]]` 링크로 감싼 단어 — 단, **참조한 `entryId`가 실재해야 한다**

3번을 마커 없이 쓰면 위반이다. "이 사전에 있는 단어니까 괜찮다"는 판단을 검증기가 대신 하지 않는다. 편집자가 명시적으로 링크를 걸어야 하고, 그래야 화면에서도 아이가 눌러서 찾아갈 수 있다.

---

## 3. 순환 정의 탐지

### 간선을 무엇으로 볼 것인가

**여기서 실수하면 검증기가 쓸모없어진다.** 정의문에 등장하는 모든 단어를 간선으로 삼으면, 기초 어휘는 원래 서로를 참조하므로 그래프가 거의 완전연결되고 모든 표제어가 순환에 걸린다.

> **간선은 `[[ ]]` 링크만이다.**
> 정의 어휘 목록에 있는 단어는 "아이가 이미 아는 단어"로 간주해 그래프에서 제외한다.

이렇게 하면 그래프가 작아지고, 남은 순환은 실제로 문제인 순환이다 — 정의 어휘 밖의 어려운 단어끼리 서로를 참조해 아이가 어디서도 출발할 수 없는 상태.

### 탐지 방법

- 노드 = 표제어(EntryId), 간선 = 정의문의 `[[ ]]` 참조
- **Tarjan SCC로 전체 강결합 요소를 찾는다.** 크기 2 이상의 SCC가 하나라도 있으면 실패
- 2단계 순환만 보지 않는다. `A→B→C→A`도 잡힌다
- 자기 참조(`A→A`)는 표제어 자기 자신 면제 규칙에 걸리므로 애초에 생기지 않는다

## 4. 검사 항목 전체

| # | 항목 | 기준 | 심각도 |
|---|---|---|---|
| V01 | 정의 어휘 위반 | 목록 밖 단어, 면제 대상 아님 | 실패 |
| V02 | 정의문 길이 | 20단어 초과 | 실패 |
| V03 | 정의문 문장 수 | 2문장 이상 | 실패 |
| V04 | 순환 정의 | SCC 크기 ≥ 2 | 실패 |
| V05 | 한국어 대응어 길이 | 4어절 이상 | 실패 |
| V06 | 예문 길이 | 6단어 미만 또는 12단어 초과 | 실패 |
| V07 | 예문 표제어 마커 | `{{ }}` 없음 | 실패 |
| V08 | 예문 어휘 | 정의 어휘 + 등재 표제어 밖 | 경고 |
| V09 | 뜻갈래 수 | 4개 이상 | 실패 |
| V10 | 뜻갈래당 예문 | 0개 | 실패 |
| V11 | `audio.us` | 없거나 파일이 실재하지 않음 | 실패 |
| V12 | `related` 참조 | 존재하지 않는 EntryId | 실패 |
| V13 | 정의 내 링크 | 존재하지 않는 EntryId | 실패 |
| V14 | id 중복 | EntryId / SenseId / ExampleId 중복 | 실패 |
| V15 | 연어의 `exampleId` | 같은 뜻갈래 안에 없음 | 실패 |
| V16 | 음절 구분점 | 구분점 제거 결과 ≠ headword | 실패 |

**V11의 파일 실재 검사**는 M1의 오디오 생성 파이프라인이 선 뒤에 켠다. 그전에는 `audio.us`가 비어 있지 않은지만 본다 — 아직 존재할 수 없는 파일을 요구하면 M0에서 모든 표제어가 실패한다.

**V08만 경고인 이유**: 예문은 자연스러움이 어휘 통제보다 우선한다. 아이가 실제로 듣는 문장이어야 하는데, 모든 단어를 1,000개 안에 가두면 문장이 부자연스러워진다. 편집자가 보고 판단한다.

**자동으로 검사하지 않는 것**: "두 예문의 문형이 서로 다른가", "예문 맥락이 아이의 생활 안에 있는가", "예문 번역이 구어체인가". 자동 판정이 불가능하므로 [편집 체크리스트](01-content-policy.md#편집-체크리스트)에 남긴다.

## 5. 테스트 데이터

| 파일 | 역할 |
|---|---|
| `data/entries.sample.json` | **전 항목 통과**하는 유효 샘플. 검증기가 이걸 실패시키면 검증기 버그 |
| `data/entries.invalid.json` | 위반 fixture. 각 표제어가 **정확히 한 가지** 규칙을 어긴다 |

`entries.invalid.json`의 각 항목은 `_expect` 필드에 위반 코드(V01 등)를 적어 둔다. 검증기 테스트는 "그 코드가, 그 표제어에서, 그것만 나오는지"를 확인한다.

> `_expect`는 fixture 전용 필드다. `Dictionary` 타입에 없으며 실제 데이터에 넣지 않는다.
