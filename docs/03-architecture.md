# 03 · 기술 결정 기록

> 결정과 **그 이유**를 남긴다. 나중에 뒤집을 때 무엇을 포기하는지 알기 위해서다.

## ADR-001 · 웹 PWA로 만든다

**결정**: Vite + React + TypeScript. 브라우저에서 열고, 홈 화면에 추가해 앱처럼 쓴다.

**이유**: 사용자가 내 아이와 지인 몇 명이다. 앱스토어 심사·서명·배포 비용이 사용자 수에 비해 과하다. 디자인 반복이 가장 빠른 것도 웹이다.

**포기하는 것**: 네이티브 수준의 음성 제어, 앱스토어 노출. 둘 다 이 프로젝트에 필요 없다.

**주 기기는 PC다** (2026-07-22 확인). 물리 키보드가 있으므로 검색·철자 입력을 타이핑 전제로 설계한다. 화면은 반응형으로 두되 태블릿 터치 최적화에 먼저 투자하지 않는다.

## ADR-002 · 서버도 계정도 없다

**결정**: 백엔드 없음. 학습 기록은 **IndexedDB**에 저장한다.

**이유**: 아동 개인정보를 **수집하지 않는 것**이 가장 안전한 처리다. 서버가 없으면 유출될 것도, 지켜야 할 개인정보 처리방침도 없다. 운영 비용도 0이다.

**포기하는 것**: 기기 간 동기화, 기기 초기화 시 학습 기록 소실.
**완화**: 학습 기록 JSON 내보내기/불러오기 버튼 하나를 둔다. 동기화보다 훨씬 싸고 대부분의 경우 충분하다.

## ADR-003 · 사전 데이터는 빌드 타임 번들

**결정**: `data/entries.json`을 빌드에 포함시킨다. 런타임 API 호출 없음.

**이유**: 오프라인 완전 동작이 요구사항이다. 1,000개 표제어는 압축 시 수백 KB 수준이라 전량 번들이 현실적이다. 검색도 클라이언트에서 끝난다.

**따라오는 것**: 표제어 하나를 고쳐도 재배포가 필요하다. 개인 프로젝트에서는 문제되지 않는다.

**검색**: 1,000개 규모에서는 라이브러리 없이 선형 탐색으로 충분하다. 성능 문제가 실제로 관찰되기 전에는 검색 엔진을 넣지 않는다. 필요한 건 오타 관용(`elefant` → `elephant`)이고, 이건 인덱스가 아니라 편집거리 계산의 문제다.

## ADR-004 · 발음 오디오는 빌드 타임에 생성해 번들한다

**결정**: 표제어·예문 음성을 **빌드 시 TTS로 생성해 mp3로 번들**한다. Web Speech API는 대체 경로로만 둔다.

**이유**: **P2(발음을 모른다)가 1순위 문제**인데, Web Speech API는 음질과 목소리가 OS·브라우저마다 제각각이라 품질을 보장할 수 없다. 발음이 제품의 핵심이면 발음을 런타임 환경에 맡길 수 없다.

기존 사전의 오디오 파일은 라이선스 문제로 쓸 수 없으므로 직접 생성한다. 1,000단어 + 예문 규모의 TTS 생성 비용은 개인 프로젝트에서 감당 가능한 수준이다.

**포기하는 것**: 번들 용량 증가, 표제어 추가 시 생성 단계 필요.
**대체 경로**: 오디오 파일이 없는 표제어는 Web Speech API로 재생한다 (`src/audio.ts`).

**재생 실패를 전제로 설계한다.** 자동 재생은 브라우저 정책·무음 모드·이어폰 미연결로 실패할 수 있다. 성공 여부와 무관하게 "다시 듣기" 버튼과 텍스트 피드백을 항상 보장한다. → [02-learning-design](02-learning-design.md)

### TTS 엔진과 라이선스 (확정, 2026-07-22)

**엔진: Piper** (`rhasspy/piper`, MIT 라이선스). **음성: en_US-lessac-medium.**

| 항목 | 값 |
|---|---|
| 엔진 라이선스 | MIT |
| 음성 모델 | en_US-lessac-medium — LESSAC 데이터 기반, 재배포 허용 |
| 실행 | 완전 오프라인, 로컬. **API 키·클라우드 계정 불필요** |
| 출력물 | 생성한 mp3는 자유롭게 사용·배포 가능 |

**왜 Piper인가**: 클라우드 TTS(Google/Amazon/Azure)는 품질은 좋지만 API 키·과금·계정이 필요하고, 개인정보 미수집 원칙(ADR-002)과 결이 다르다. Piper는 로컬에서 돌고, 출력물 재배포가 자유로우며, 음질이 초등 발음 학습에 충분하다. eSpeak은 음질이 로봇 같아 P2에 부적합해 배제했다.

**파이프라인** (`tools/audio/generate.py`): 음성 모델 1회 로드 → 표제어별 WAV 합성 → ffmpeg(imageio-ffmpeg 정적 바이너리)로 mp3 인코딩 → `public/audio/us/<id>.mp3`. 증분 생성(있으면 건너뜀). `npm run audio`.

**용량**: 132개 = 459 KB (표제어당 평균 ~3.5 KB, 22 kHz mono VBR). 생성 mp3는 커밋한다 — 파이썬 파이프라인(+61MB 음성 모델) 없이도 앱이 발음을 내야 하기 때문. 음성 모델 자체는 재다운로드 가능하므로 커밋하지 않는다.

**검증**: `npm run validate:audio`가 V11로 모든 `audio.us` 파일의 실재를 확인한다.

- **어휘 목록**: 교육부 고시 제2022-33호 [별책 14] 영어과 교육과정, [별표 3] 기본 어휘 (pp. 254–290). 한국교육과정평가원 KICE Word Lister 자료집을 통해 확보(2026-07-21). 추출본은 `data/moe-vocabulary.json`이며 `_source`에 출처가 함께 저장되어 있다

## ADR-005 · 미국식 발음을 기본으로 한다

**결정**: 화면 기본은 US 발음. UK는 확연히 다를 때만 병기.

**이유**: 한국 초등 교과서와 교실 음성 자료가 미국식이다. 학교에서 듣는 소리와 앱의 소리가 다르면 아이가 혼란스럽다. 상세 근거는 [01-content-policy](01-content-policy.md) 5절.

**반영 완료**: `src/types/entry.ts`(`ipa.us` 필수, `ipa.uk` 선택), `prototype/entry.html`(US 우선, `en-US` 재생).

## ADR-007 · 복습 예정일은 로컬 날짜 문자열로 저장한다

**결정**: `StudyRecord.dueOn`은 ISO 타임스탬프가 아니라 **로컬 자정 기준 `'YYYY-MM-DD'`** 문자열.

**이유**: 아이에게 '오늘 할 일'은 시각이 아니라 날짜다. 타임스탬프로 저장하면 여행이나 서머타임 전환 때 하루가 사라지거나 두 번 생긴다. 통계용 `lastAnsweredAt`은 ISO 8601로 두되 **스케줄링에는 쓰지 않는다.**

**왜 지금 정하는가**: 전이 규칙과 달리 이건 저장된 데이터의 형식이라 나중에 바꾸면 기존 기록을 마이그레이션해야 한다.

## ADR-008 · 버전 필드를 처음부터 넣는다

**결정**: 세 곳에 버전을 둔다.

| 필드 | 위치 | 용도 |
|---|---|---|
| `Dictionary.schemaVersion` | 사전 데이터 | 구조 변경 감지 |
| `Dictionary.contentVersion` | 사전 데이터 | 서비스 워커 캐시 갱신 판단 |
| `StudyState.schemaVersion` | IndexedDB | 학습 기록 마이그레이션 |

**이유**: 버전 필드는 **나중에 소급 삽입이 불가능하다.** 이미 저장된 데이터에는 버전이 없으므로 "버전 없음 = v0"이라는 추가 규칙을 영구히 안고 가야 한다. 지금 넣는 비용은 필드 세 개다.

**`StudyState`에 버전을 두는 이유** (레코드마다가 아니라): 마이그레이션 진입점이 하나여야 부분 마이그레이션 상태가 생기지 않는다.

**M1/M2로 미루는 것**: 서비스 워커 캐시 갱신 전략(M1), 학습 기록 가져오기의 유효성 검사·충돌 처리(M2). 둘 다 코드만 고치면 되돌릴 수 있다. 단 가져오기에는 **깨진 파일이 기존 기록을 덮어쓰지 않는다**는 규칙을 반드시 넣는다 — 데이터 손실은 되돌릴 수 없다.

**`sense.id` 삭제·병합 시 마이그레이션**: sense id는 재사용하지 않으므로, 사라진 id를 참조하는 학습 기록은 **조용히 폐기한다.** 다른 뜻갈래로 옮겨 붙이지 않는다 — 아이가 학습한 뜻과 다른 뜻이 될 수 있다.

## 데이터 모델

`src/types/entry.ts`에 정의되어 있다. 핵심 결정 세 가지:

1. **난이도는 뜻갈래(sense) 단위** — 학년별로 사전을 복제하지 않는다
2. **학습 기록은 `sense.id`를 참조** — 표제어 단위로는 "어떤 뜻을 아는지" 구분이 불가능하다. 그래서 sense id는 한 번 발급하면 **재사용하지 않는다**
3. **표제어 위치는 인덱스가 아니라 `{{ }}` 마커** — 예문을 수정해도 위치가 어긋나지 않는다
4. **단어를 가리키는 모든 참조는 `EntryId`** — 관련어와 정의 내 링크. `light`라는 문자열만으로는 `light-1`(빛)과 `light-3`(가벼운)을 구분할 수 없다. 동형이의어가 하나라도 생기면 문자열 참조는 전량 재작업이 된다
5. **`tags`는 뜻갈래 단위** — 학습이 뜻갈래 단위이므로. 표제어에 붙이면 `run`의 '달리다'와 '작동하다'가 같은 주제로 묶여 오답 보기가 엉뚱해진다
